# 코드 플래시카드 생성 스킬

사용자가 소스 코드(또는 코드가 포함된 자료)를 제공하면, 설계 문제점을 분석하고 학습용 플래시카드 MD 파일을 생성한다.

### 2단계: Source 분석

사용자가 제공한 소스 코드에서 다음을 파악한다:

- **설계 냄새(Design Smell)**: 어떤 원칙이 위반되고 있는가?
  - 변경의 이유가 여러 개인가? (SRP 위반)
  - 구현에 직접 의존하고 있는가? (DIP 위반)
  - 확장하려면 기존 코드를 수정해야 하는가? (OCP 위반)
  - 교체 가능한 구조인가? (LSP/다형성)
- **변경 시나리오**: "만약 ~해야 한다면?" 형태의 현실적인 상황을 2-3개 도출
- **적용할 패턴/원칙**: Strategy, Observer, DI, 인터페이스 분리 등

## 작성 원칙

### 질문부
- **코드 먼저, 질문 나중**: 실제 코드를 보여준 뒤 설계 문제를 묻는다
- **서브 질문 순서**: 구체적→추상적 (변경 요구→문제 인식→패턴 적용)
- **힌트**: "변경의 이유"나 "책임"처럼 핵심 개념을 암시하되 답을 주지 않는다

### 답변부
- **핵심 아이디어**: 한 단락, 본질적인 "왜"에 집중
- **코드는 역할별로 분리**: 한 번에 전체를 보여주지 않고 단계별로 제시
- **코드 내 주석**: 설계 의도를 드러내는 짧은 주석 (`// 의도: ~`)
- **"왜?" 섹션**: 반드시 포함. "SOLID이니까"가 아니라 실제 시나리오로 설명
  - "react-router를 Next.js로 바꿀 때 storage 로직을 건드려야 하는 상황이 생긴다" 같은 구체성

### 톤앤매너
- **한국어 기반**, 기술 용어(interface, Strategy Pattern 등)는 영어 유지
- **설명체**: "~하다", "~이다" (대화체 아님)
- 코드 블록에는 파일명 주석 포함 (`// usePagination.ts`)
- 핵심 키워드는 **볼드** 처리

### 난이도 기준
- **easy**: 단일 원칙 적용 (SRP만, DIP만)
- **medium**: 2-3개 원칙 조합 + 패턴 (SRP + DI + Strategy)
- **hard**: 복합 패턴 + 트레이드오프 분석 필요

### 태그
- 적용된 설계 원칙/패턴을 태그로 (4개 이내)

### 파일명
- `{3자리 ID}-{kebab-case-주제}.md`

## 주의사항

- 소스 코드가 이미 잘 설계된 경우: 의도적으로 "나쁜 버전"을 질문으로 제시하고 현재 코드를 답변으로 활용할 수 있다
- 프레임워크 특정 코드: React/TS 기준이지만, 설계 원칙은 프레임워크에 구애받지 않는다. 원칙 중심으로 설명한다
- 코드가 너무 길면: 핵심 문제가 드러나는 부분만 발췌하여 질문에 넣는다
- 사용자가 특정 패턴이나 원칙을 지정하면 그에 맞춰 카드를 구성한다
