## 질문

Qwen3.5 9B 모델을 다양한 비트로 양자화한 후 품질을 측정했다. 아래 코드와 결과를 분석하시오.

```javascript
// 퍼플렉서티(perplexity) 계산
function measurePerplexity(model, testTokens) {
    let total = 0;
    for (const { correctToken, probDistribution } of slidingWindow(model, testTokens)) {
        const pCorrect = probDistribution[correctToken];
        total += -Math.log(pCorrect);
    }
    return Math.exp(total / testTokens.length);
    // 낮을수록 좋다: "모델이 후보가 N개라고 생각한다"
}

// KL 발산(KL Divergence) 계산
function measureKLDivergence(originalDist, quantizedDist) {
    let kl = 0;
    for (const token of allTokens) {
        const p = originalDist[token]; // 원본 확률
        const q = quantizedDist[token]; // 양자화 확률
        if (p > 0) kl += p * Math.log(p / q);
    }
    return kl;
    // 0에 가까울수록 두 분포가 비슷하다
}
```

```
측정 결과 (Qwen3.5 9B):

│ 포맷             │ 퍼플렉서티 │ KL 발산      │ GPQA 정답률  │
├──────────────────┼────────────┼──────────────┼──────────────┤
│ bfloat16 (원본)  │ 8.186      │ -            │ 66.7%        │
│ 8비트 대칭       │ 8.193      │ 0.0008       │ 73.2%        │
│ 4비트 비대칭     │ 8.563      │ 0.0593       │ 62.6%        │
│ 4비트 대칭       │ 8.71       │ 0.0675       │ 66.2%        │
│ 2비트 비대칭     │ 66.1       │ 2.1447       │ 1%           │
└──────────────────┴────────────┴──────────────┴──────────────┘
```

1. 퍼플렉서티가 `8.186 → 8.563`으로 오르는 것과 `8.186 → 66.1`로 오르는 것의 **질적 차이**를 설명하시오. 퍼플렉서티 66.1은 실제로 무엇을 의미하는가?
2. 퍼플렉서티는 **정답 토큰의 확률만** 측정한다. KL 발산이 추가로 포착할 수 있는 것은 무엇인가? 구체적 예시를 들어 설명하시오.
3. 8비트 → 4비트는 퍼플렉서티가 **선형적으로** 증가하지만, 4비트 → 2비트에서 **급격한 절벽(cliff)**이 발생한다. 이 비선형적 품질 저하의 원인을 추론하시오.

**힌트:** 양자화 비트가 줄어들면 표현 가능한 고유 값의 수가 지수적으로 감소한다 (8비트: 256개, 4비트: 16개, 2비트: 4개).

## 정답: 양자화 품질 절벽 (Quality Cliff)

### 핵심 아이디어

양자화 비트 수를 줄이면 품질이 **선형이 아닌 비선형적으로** 저하된다. 특히 2비트(4개의 고유 값)에서는 파라미터의 미세한 차이를 구분할 수 없어 모델이 **사실상 붕괴**한다. 이 "절벽"을 식별하는 도구가 퍼플렉서티와 KL 발산이다.

### 각 질문 해설

**1. 퍼플렉서티의 의미**

```javascript
// 퍼플렉서티 8.186 (원본)
// → "다음 토큰 후보가 ~8개 정도라고 느낀다"
// → 모델이 꽤 확신을 가지고 있다

// 퍼플렉서티 8.563 (4비트)
// → 후보가 ~8.5개 정도 → 약간 덜 확신
// → 실사용에서 거의 차이 없음 (+4.6%)

// 퍼플렉서티 66.1 (2비트)
// → "다음 토큰 후보가 ~66개라고 느낀다"
// → 거의 무작위에 가까운 수준
// → 실제 행동: 같은 문장을 반복하다 빈 응답 출력

// 2비트 모델의 실제 추론 과정:
// "The capital city of England is London.
//  The capital city of England is London.
//  The capital city of England is London.
//  ..." (무한 반복 후 빈 응답)
```

**2. 퍼플렉서티 vs KL 발산**

```javascript
// 예: 프롬프트 "The answer to 2 + 2 is"
// 원본 분포:    { "4": 0.92, "5": 0.03, "3": 0.01, ... }
// 양자화 분포A: { "4": 0.92, "5": 0.01, "3": 0.05, ... }
// 양자화 분포B: { "4": 0.92, "2": 0.04, "7": 0.03, ... }

// 퍼플렉서티: 정답 "4"의 확률(0.92)만 봄
// → 분포 A와 B를 구분 못 함! 둘 다 퍼플렉서티 동일

// KL 발산: 전체 분포의 차이를 측정
// → 분포 B는 "2"와 "7"에 확률이 몰려 있어
//   원본과 더 다르다는 것을 포착
const klA = measureKLDivergence(original, distA); // 낮음
const klB = measureKLDivergence(original, distB); // 높음
```

이것이 중요한 이유: LLM의 생성은 **샘플링** 기반이다. 정답 토큰의 확률이 같아도, 2순위/3순위 토큰의 분포가 바뀌면 **다양성, 일관성, 추론 체인**이 달라진다.

**3. 비선형적 품질 절벽의 원인**

```javascript
// 표현 가능한 고유 값 수
const uniqueValues = {
    "8비트": 2 ** 8,  // 256개
    "4비트": 2 ** 4,  // 16개
    "2비트": 2 ** 2,  // 4개 ← 여기서 절벽!
};

// 4비트 → 16개 값으로 구분
// 파라미터 범위 -0.89 ~ 0.16을 16개 "버킷"으로 나눔
// 각 버킷 폭: (0.89 + 0.16) / 16 ≈ 0.066
// → 0.08과 0.16을 서로 다른 버킷에 배치 가능

// 2비트 → 4개 값으로 구분
// 각 버킷 폭: (0.89 + 0.16) / 4 ≈ 0.263
// → 0.08, 0.16, -0.13이 전부 같은 버킷에 뭉개짐
// → 파라미터 간 "상대적 차이"가 소멸
```

| 비트 | 고유 값 | 버킷 폭 (예시) | 구분 가능한 차이 |
|------|--------|---------------|----------------|
| 8 | 256 | 0.004 | 매우 미세한 차이 |
| 4 | 16 | 0.066 | 실용적 수준 |
| 2 | 4 | 0.263 | 대부분 구분 불가 |

### 깊은 이유 설명

품질 절벽의 근본 원인은 **정보 이론적 한계**다. LLM의 파라미터들은 서로 다른 값을 가짐으로써 **정보를 인코딩**한다. 양자화는 이 고유 값의 수를 줄이는 것이고, 어느 임계점 아래로 내려가면 **모델이 인코딩한 정보 자체가 파괴**된다.

8비트(256개 값)와 4비트(16개 값)의 차이는 "정밀도 손실"이다. 값들이 약간 부정확해지지만, 파라미터 간의 **상대적 순서와 비율**은 대체로 보존된다. 하지만 2비트(4개 값)에서는 수십억 개의 파라미터가 **단 4개의 값으로** 표현되어야 한다. 이는 더 이상 "부정확한 표현"이 아니라 **"구분 불가능"**이다.

이것이 양자화 결과가 선형이 아닌 이유다:
- **16비트 → 8비트**: 품질 손실 ~0.1% (거의 무손실)
- **16비트 → 4비트**: 품질 손실 ~5-10% (실용적)
- **16비트 → 2비트**: 품질 손실 ~700% (완전 붕괴)

> 실무 가이드: 로컬에서 LLM을 돌릴 때, **Q4 계열(Q4_0, Q4_1)**이 크기와 품질의 최적 균형점이다. Q8은 품질은 최고지만 크기가 크고, Q2는 크기는 작지만 사용 불가 수준이다. "절벽에서 떨어지지 않는 선에서 가장 작은 포맷"을 선택하는 것이 핵심이다.
