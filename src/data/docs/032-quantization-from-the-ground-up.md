## 왜 양자화가 필요한가?

[Qwen-3-Coder-Next](https://huggingface.co/Qwen/Qwen3-Coder-Next)는 800억 개의 파라미터를 가진 모델로, 크기가 **159.4GB**에 달한다. 이걸 실행하려면 대략 그만큼의 RAM이 필요하고, 긴 컨텍스트 윈도우까지 고려하면 더 많이 필요하다. *그런데 이건 큰 모델도 아니다.* 프론티어 모델들은 1조 개가 넘는 파라미터를 가지고 있다는 소문이 있고, 그러면 최소 **2TB**의 RAM이 필요하다.

하지만 LLM을 **4배 작게, 2배 빠르게** 만들면서 정확도 손실은 **5~10%** 수준으로 유지할 수 있다면?

이것이 바로 **양자화(quantization)**의 마법이다.

---

## 1. 대규모 언어 모델은 왜 그렇게 큰가?

### 파라미터(가중치)란?

파라미터(parameter), 또는 가중치(weight)는 LLM이 메모리나 디스크에 있을 때 그 대부분을 차지하는 것이다. LLM은 "수십억 개의 신중하게 배치된 연산으로 이루어진 거대한 그래프"다.

가장 단순한 예시부터 보자: **입력 1개, 파라미터 1개, 출력 1개**.

```
입력: 2.0  →  [× 0.5]  →  출력: 1.0
```

별거 아닌 것 같지만, 이것이 현대 AI의 기본 빌딩 블록이다. 입력 `2.0`에 파라미터 `0.5`를 곱해서 출력 `1.0`을 얻는다.

### 레이어(층)

LLM이 거대해지는 방법 중 하나는 "레이어"를 쌓는 것이다.

```
입력: 2.0
    ├─ [× 0.5] → 중간값 1.0 ─┬─ [× 1.0] ─┐
    └─ [× 0.5] → 중간값 1.0 ─┤            ├→ 출력: 1.5
                              └─ [× 0.5] ─┘
```

두 연결이 같은 노드에서 끝나면, 그 값들을 **더한다**. 출력 `1.5`는 `1.0 × 1.0`과 `0.5 × 1.0`을 더한 것이다.

> 🎮 원문에서 입력값 슬라이더를 움직여보면 출력이 어떻게 변하는지 직관적으로 이해할 수 있다.

이 예시는 겨우 파라미터 6개다. 현대 LLM은 수십만 개의 입력과 출력, 수십 개의 레이어, 각 레이어에 수천 개의 노드가 밀집 연결(densely connected)되어 있다. 이 모든 게 곱해져서 **수십억**, 때로는 **수조 개**의 파라미터가 된다.

## 2. 컴퓨터는 숫자를 어떻게 저장하는가?

### 정수(integer)

컴퓨터는 `1`과 `0`(비트)으로 동작한다. 정수는 이렇게 저장된다:

```
비트:     0  1  0  0  0  0  0  0
2의 거듭제곱: 128 64  32  16  8  4  2  1
              → 64
```

각 비트가 2의 거듭제곱 숫자를 나타내고, 이걸 합하면 최종 값이 된다.

> 🎮 원문에서 비트를 직접 클릭하며 토글해볼 수 있다.

정수는 **이산적(discrete)**이라 다루기 편하다. 1과 3 사이에 정수는 딱 하나, 2뿐이다.

### 소수점의 문제

1과 3 사이에 소수점 수는 몇 개인가? **무한 개**다. 컴퓨터는 무한한 것을 표현할 수 없다.

그래서 컴퓨터는 **타협**한다. 일정한 유효 숫자(significant figures)까지만 정확하게 표현하고, 그 이후는 최선의 근사값을 쓰기로 한다.

### 부동소수점(floating point)

32비트 부동소수점(float32)은 ±3.40×10³⁸ 범위를 **유효 숫자 7자리** 정확도로 표현한다. 32비트를 3부분으로 나눈다:

| 구성 요소 | 비트 수 | 역할 |
|-----------|--------|------|
| 부호(sign) | 1비트 | 양수/음수 |
| 지수(exponent) | 8비트 | 범위(얼마나 크거나 작은 수를 표현할 수 있는가) |
| 가수(significand) | 23비트 | 정밀도(유효 숫자 몇 자리까지 정확한가) |

지수 비트가 많으면 **더 넓은 범위**, 가수 비트가 많으면 **더 높은 정밀도**를 얻는다.

> 🎮 원문에서 float32 시각화의 +/- 버튼을 눌러보면, "다음으로 표현 가능한 값"으로 이동한다. 유효 숫자 이후의 자릿수가 가끔 건너뛰는 것을 확인할 수 있다 — 이것이 정밀도 타협의 실체다. 아주 작은 값에서 +를 누르면 조금만 이동하고, 아주 큰 값에서 +를 누르면 크게 이동한다. **값이 균등하게 분포하지 않는다.**

### float32 값의 분포

32비트 부동소수점으로 표현 가능한 값의 히스토그램을 보면, **-0.5~0.5 사이에 20억 개 이상의 고유 값**이 몰려 있다. 작은 값 쪽에 표현 가능한 수가 엄청나게 많다.

이건 LLM에 아주 좋은 소식이다. **파라미터 값도 대부분 작기 때문**이다. 작은 파라미터는 모델이 학습 시 본 적 없는 문제에도 잘 일반화(generalize)하게 해주므로, 학습 과정에서 파라미터가 작아지도록 보상을 받는다.

실제로 6개의 유명 오픈소스 모델의 파라미터 값 분포를 보면, **거의 모든 파라미터가 0에 매우 가깝다.**

> 🎮 원문의 모델 파라미터 분포 히스토그램에서 직접 확인할 수 있다.

## 3. 더 작은 부동소수점을 쓸 수 있을까?

언어 모델이 정말 32비트 부동소수점이 **필요한가**? 넓은 범위도 필요 없고, 유효 숫자 7자리의 정확도가 정말 필요한가?

**답은 NO**다. LLM은 더 작고 덜 정확한 부동소수점으로도 잘 작동한다.

### float16 (16비트)

| 구성 | 비트 수 | 성능 |
|------|--------|------|
| 부호 | 1 | - |
| 지수 | 5 | 범위: ±65,504 |
| 가수 | 10 | 유효 숫자: 3자리 |

float32의 **절반 크기**의 RAM과 디스크를 차지한다.

### bfloat16 (Brain Float 16)

구글 브레인 팀이 만든 포맷이다.

| 구성 | 비트 수 | 성능 |
|------|--------|------|
| 부호 | 1 | - |
| 지수 | 8 | 범위: ±3.39×10³⁸ (매우 넓음!) |
| 가수 | 7 | 유효 숫자: 2자리 |

구글은 **유효 숫자 2자리면 LLM을 만들기에 충분**하다는 것을 발견했다. 그리고 매우 넓은 범위 덕분에 큰 LLM에서 발생할 수 있는 오버플로우(overflow) 걱정을 하지 않아도 된다.

### float8, float4

더 극단적인 예시도 있다. 실무에서는 지수 비트와 가수 비트의 수를 필요에 맞게 조합해서 사용한다.

> 🎮 원문에서 사인파(sine wave)를 각 부동소수점 포맷으로 근사한 그래프를 볼 수 있다. 줌 버튼으로 1x → 30x → 200x로 확대하면 차이가 극명하게 드러난다.

## 4. 양자화란 무엇인가?

양자화(quantization)는 **큰 범위의 값을 작은 범위로 압축하는 과정**이다. **손실 압축(lossy compression)**의 일종이다.

예를 들어 float16에서 float8로 변환할 때, "가장 가까운 표현 가능한 값"으로 반올림한다. 이를 "round-to-nearest(가장 가까운 값으로 반올림)"이라 하며, 양자화의 여러 유형 중 하나다.

### 단순 반올림의 문제

작은 모델의 파라미터를 bfloat16에서 float8, float4로 단순 반올림하면 어떻게 될까?

- **float8로 반올림**: 그럭저럭 괜찮다.
- **float4로 반올림**: **모델이 완전히 망가진다.** 일부 파라미터가 0이 되어버리기 때문이다. 입력에서 출력까지의 모든 경로가 0을 곱하게 되면, 출력은 항상 0이다.

이 문제는 사용 가능한 4비트를 **비효율적으로 사용**하기 때문에 발생한다. float4는 -3에서 3까지의 범위를 커버하지만, 실제 파라미터는 -0.89에서 0.16 사이에 있다. 게다가 float4는 `Infinity`와 `NaN`도 표현할 수 있는데, 양자화에서는 이것들이 아무 쓸모가 없다.

4비트가 주는 16개의 값을 더 효율적으로 쓸 방법을 알아보자.

---

## 5. 대칭 양자화 (Symmetric Quantization)

float4처럼 -3~3의 범위 대신, **데이터에 더 잘 맞는 좁은 범위**를 쓰면 어떨까?

핵심 아이디어는 데이터를 새로운 범위로 **스케일링(scaling)**하는 것이다.

### 원리

-14~14 범위의 데이터를 -7~7에 넣고 싶다면, **2로 나누면** 된다. 원래 값으로 복원하려면 **2를 곱하면** 된다. 홀수 값(예: 5)은 표현할 수 없게 되어 가장 가까운 정수로 반올림된다 (5 ÷ 2 = 2.5 → 반올림하여 3, 역양자화하면 3 × 2 = 6). 이것이 양자화가 **손실**인 이유다.

### 스케일 팩터 계산

모든 값 집합에 이 과정을 적용할 수 있다. 적절한 스케일 팩터만 찾으면 된다:

**스케일 = 데이터의 최대 절대값 / 양자화 범위의 최대값**

```javascript
function quantize({ values, bits }) {
    // values = [-0.89, 0.16, 0.08, -0.13, 0.16, -0.54]
    // bits = 4
    const vmax = Math.max(...values.map(Math.abs)); // 0.89
    const qmax = 2 ** (bits - 1) - 1;               // 7
    const scale = vmax / qmax;                       // 0.1271...
    return {
        values: values.map((v) => Math.round(v / scale)),
        scale,
    };
}

function dequantize({ values, scale }) {
    return values.map((v) => v * scale);
}
```

### 적용 예시

```javascript
const values = [-0.89, 0.16, 0.08, -0.13, 0.16, -0.54];
const quantized = quantize({ values, bits: 4 });
// { values: [-7, 1, 1, -1, 1, -4], scale: 0.1271... }
```

역양자화(dequantize)하면:

```javascript
const dequantized = dequantize(quantized);
// [-0.89, 0.1271, 0.1271, -0.1271, 0.1271, -0.5086]
```

### 오차 비교

| 원본 | 역양자화 값 | 오차 % |
|------|-----------|--------|
| -0.89 | -0.89 | 0.0% |
| 0.16 | 0.1271 | -20.6% |
| 0.08 | 0.1271 | +58.9% |
| -0.13 | -0.1271 | -2.2% |
| 0.16 | 0.1271 | -20.6% |
| -0.54 | -0.5086 | -5.8% |
| **평균 오차** | | **18.0%** |

모델을 **4배 작게** (16비트 → 4비트) 만들었는데, 역양자화 값의 평균 오차가 약 18%다. 나쁘지 않다!

모델에 적용하면, 최종 출력이 약 **30% 정도 차이**가 난다. 항상 0을 출력하던 단순 반올림 float4에 비하면 엄청난 개선이다. 특히 이 양자화된 모델은 원본의 **1/4 RAM**만 필요하다는 걸 기억하자.

하지만... 더 잘할 수 있다.

## 6. 비대칭 양자화 (Asymmetric Quantization)

대칭 양자화는 **0이 항상 중앙**에 있다. 스케일을 정해서 0을 중심으로 값을 압축한다.

문제는 양쪽 범위가 같다는 것이다. 최대값이 0.16이고 최소값이 -0.89인데, 대칭 양자화는 양쪽 모두 -0.89~0.89 범위를 잡는다. 양의 쪽에 0.16~0.89까지 쓸데없는 공간이 낭비된다.

### 핵심 아이디어

비대칭 양자화는 **불균등한 범위를 허용**한다. 0 중심이 아니라 **데이터의 중간점(midpoint)**을 중심으로 압축하고, 역양자화 시 보정할 "영점(zero point)"을 별도로 저장한다.

```javascript
function quantize({ values, bits }) {
    const vmax = Math.max(...values);            // 0.16
    const vmin = Math.min(...values);            // -0.89
    const qmax = 2 ** (bits - 1) - 1;           // 7
    const qmin = -(2 ** (bits - 1));             // -8
    const scale = (vmax - vmin) / (qmax - qmin); // 0.07
    const zero = qmin - Math.round(vmin / scale); // 5
    return {
        values: values.map((x) => Math.round(x / scale + zero)),
        scale,
        zero,
    };
}
```

### 적용 예시

```javascript
const values = [-0.89, 0.16, 0.08, -0.13, 0.16, -0.54];
const quantized = quantize({ values, bits: 4 });
// { values: [-8, 7, 6, 3, 7, -3], scale: 0.07, zero: 5 }
```

역양자화에는 뺄셈이 추가된다:

```javascript
function dequantize({ values, scale, zero }) {
    return values.map((x) => scale * (x - zero));
}
// [-0.91, 0.14, 0.07, -0.14, 0.14, -0.56]
```

### 오차 비교

| 원본 | 역양자화 값 | 오차 % |
|------|-----------|--------|
| -0.89 | -0.91 | +2.2% |
| 0.16 | 0.14 | -12.5% |
| 0.08 | 0.07 | -12.5% |
| -0.13 | -0.14 | +7.7% |
| 0.16 | 0.14 | -12.5% |
| -0.54 | -0.56 | +3.7% |
| **평균 오차** | | **8.5%** |

훨씬 낫다! 같은 비트 수에서 **오차가 10% 포인트나 줄었다.**

> 🎮 원문에서 bfloat16과 양자화된 4비트 모델을 토글하면서 비교해볼 수 있다.

### 실제로 일어나는 일

모델이 양자화되어 노트북에서 실행 가능한 크기가 될 때, 부동소수점 대신 **작은 정수**가 저장되고 메모리에 로드된다. 질문에 대한 답을 생성할 때 등 양자화된 값을 사용해야 할 때, 값이 **실시간으로 역양자화**된다. 느릴 것 같지만, 나중에 보겠지만 실제로는 **더 작을 뿐 아니라 더 빠르기도** 하다.

## 7. 실무에서 양자화는 어떻게 적용되는가?

수천억 개의 파라미터를 가진 LLM에서 전체의 최대/최소를 찾아서 한 번에 양자화할까?

**아니다.**

### 이상치(outlier) 문제

대부분의 모델에는 소수의 **이상치 파라미터**가 있다. 다른 파라미터들보다 훨씬 크거나 작은 값들이다. 이상치는 양자화에 **치명적**이다.

예를 들어 기존 파라미터에 `10`이라는 이상치 하나만 추가하면:

| 원본 | 역양자화 값 | 오차 % |
|------|-----------|--------|
| -0.89 | 0.726 | -181.6% |
| 0.16 | 0 | -100.0% |
| 0.08 | 0 | -100.0% |
| -0.13 | 0 | -100.0% |
| 0.16 | 0 | -100.0% |
| -0.54 | 0.726 | -234.4% |
| 10 | 10.164 | +1.6% |
| **평균 오차** | | **116.8%** |

모든 것이 소수의 버킷으로 쭈그러들고 평균 오차가 폭등한다.

### 블록 단위 양자화

전체 모델을 한 번에 양자화하면 모델이 망가진다. **실무에서는 보통 32~256개 파라미터 단위의 블록으로 양자화**한다. 이렇게 하면 이상치의 영향이 해당 블록에 국한된다.

역양자화를 위해 각 블록마다 `scale` (대칭) 또는 `scale` + `zero` (비대칭) 값을 함께 저장해야 한다. 이것은 **오버헤드**다. 블록 크기가 클수록 오버헤드는 줄지만, 큰 블록은 평균적으로 값의 범위가 넓어져 오차가 증가한다. **트레이드오프**다.

### "이상치는 왜 존재하는가?"

정확한 이유는 아무도 모르지만, 이 이상치 중 극소수는 모델 품질에 **매우 중요**하다. Apple이 "슈퍼 가중치(super weight)"라고 부르는 것 중 **단 하나만 제거해도** 모델이 완전한 헛소리를 출력할 수 있다.

이 때문에 실제 양자화 방식에서는 이상치를 보존하기 위한 추가 작업을 한다:
- 이상치를 아예 양자화하지 않거나
- 위치와 값을 별도 테이블에 저장한 후 제거하여 해당 블록이 망가지지 않게 하고, 역양자화 시 테이블을 참조해서 이상치를 복원한다

## 8. 양자화가 모델 정확도에 미치는 영향

아래 모든 테스트는 [Qwen3.5 9B](https://huggingface.co/Qwen/Qwen3.5-9B) 모델에 대해 수행되었다.

### 8-1. 퍼플렉서티 (Perplexity)

LLM은 주어진 프롬프트에 대해 다음 "토큰"의 **확률 분포**를 생성한다.

예: 프롬프트 `The answer to 2 + 2 is`에 대한 확률:

| 토큰 | 확률 |
|------|------|
| 4 | 92.29% |
| 5 | 3.23% |
| 3 | 1.15% |
| 1 | 0.90% |
| 2 | 0.85% |

퍼플렉서티는 이 확률 분포를 하나의 숫자로 요약한다:

```javascript
pCorrect = 0.9229; // 정답 토큰 '4'의 확률
perplexity = Math.exp(-Math.log(pCorrect)); // 1.08
```

**낮을수록 좋다.** "모델이 이 프롬프트를 완성할 수 있는 후보가 ~1.08개 정도라고 생각한다"로 읽으면 된다.

더 모호한 프롬프트 `And then I`의 경우, 확률이 여러 토큰에 분산된다 (have: 3.02%, was: 3.00%, realized: 2.98%...). 정답을 `was`로 가정하면:

```javascript
pCorrect = 0.03;
perplexity = Math.exp(-Math.log(pCorrect)); // 33.33
```

훨씬 높지만, "And then I" 다음에 뭐가 올지 당신도 못 맞출 것이다.

#### 측정 방법

`llama.cpp`의 `llama-perplexity` 도구를 사용한다. 참조 텍스트 위로 **슬라이딩 윈도우**를 이동하면서, 다음 토큰이 무엇이어야 하는지 알고 있는 상태에서 모델의 예측 확률을 수집한다.

> 🎮 원문에서 슬라이딩 윈도우가 토큰 위를 이동하는 것을 시각적으로 볼 수 있다.

최종 계산:

```javascript
let total = 0;
for (const prob of probs) {
    total += -Math.log(prob);
}
const perplexity = Math.exp(total / probs.length);
```

#### 결과

| 포맷 | 퍼플렉서티 |
|------|-----------|
| bfloat16 (원본) | 8.186 |
| 8비트 대칭 | 8.193 (+0.1%) |
| 4비트 비대칭 | 8.563 (+4.6%) |
| 4비트 대칭 | 8.71 (+6.4%) |
| 2비트 비대칭 | 66.1 (+707.5%) |

8비트는 거의 변화가 없고, 4비트에서 약간의 품질 저하, 2비트에서는 거의 완전한 붕괴가 발생한다.

**퍼플렉서티의 한계**: 정답 토큰의 확률**만** 본다. 다른 모든 토큰의 확률 변화는 포착하지 못한다.

### 8-2. KL 발산 (KL Divergence)

"쿨백-라이블러 발산"이라고 하며, **두 확률 분포가 얼마나 겹치는지** 측정한다.

> 🎮 원문에서 두 개의 정규분포를 슬라이더로 이동시키며 KL 발산 값이 변하는 것을 체험할 수 있다. 두 분포가 정확히 겹칠 때만 KL 발산이 0이다.

수평 이동(skew)만 KL 발산을 높이는 게 아니라, **어떤 형태의 비겹침이든** 값을 높인다 (예: 분포의 높이/너비가 달라지는 것도).

> 🎮 원문에서 `The answer to 2 + 2 is`에 대한 토큰 확률 분포를 양자화 수준별로 토글하면서 KL 발산 변화를 볼 수 있다.

#### 결과

| 포맷 | 평균 KL 발산 |
|------|-------------|
| 8비트 대칭 | 0.0008 |
| 4비트 비대칭 | 0.0593 |
| 4비트 대칭 | 0.0675 |
| 2비트 비대칭 | 2.1447 |

KL 발산의 단점: 직관적으로 해석하기 어렵고, 자연적인 최대값도 없다. **같은 모델의 양자화 간 비교**에서만 유의미하다.

KL 발산의 장점: 퍼플렉서티와 달리 **전체 확률 분포**를 고려한다. 정답 토큰의 확률이 같아도 다른 토큰들의 분포가 바뀌면 그것을 포착한다.

### 8-3. 벤치마크

[GPQA Diamond](https://github.com/idavidrein/gpqa) 벤치마크를 실행했다. 생물학, 화학, 물리학의 매우 어려운 198개 객관식 문제(4지선다)다. 무작위 추측은 평균 25% 정답률을 낸다.

| 포맷 | 정답 | 오답 | 무응답 |
|------|------|------|--------|
| bfloat16 | 66.7% | 33.3% | 0% |
| 8비트 대칭 | 73.2% | 26.8% | 0% |
| 4비트 비대칭 | 62.6% | 36.4% | 1% |
| 4비트 대칭 | 66.2% | 29.3% | 4.5% |
| 2비트 비대칭 | 1% | 2% | 97% |

8비트가 원본보다 **더 높은 점수**를 받는 놀라운 결과가 나왔다. 정확한 이유는 불분명하지만, 객관식의 25% 우연 확률 때문일 수 있다. 핵심은 8비트와 4비트는 잘 작동하지만, **2비트는 완전히 무너졌다**는 것이다 (97% 무응답).

### 8-4. 그냥 대화해보기

가장 쉽지만 가장 덜 엄밀한 테스트: 직접 말을 걸어보는 것이다. "What is the capital city of England, UK?"라는 질문을 했다.

| 포맷 | 답변 |
|------|------|
| bfloat16 원본 | **London** ✅ |
| 8비트 대칭 | **London** ✅ |
| 4비트 비대칭 | **London** ✅ |
| 4비트 대칭 | **London** ✅ |
| 2비트 비대칭 | *<무응답>* ❌ |

2비트 양자화는 거의 매번 답변을 거부했다. "추론 과정"을 보면:

```
The capital city of England is London.
The capital city of England is London.
The capital city of England is London.
(... 반복 ...)
The capital city of London
The capital city of London
(... 반복 ...)
```

이렇게 반복하다가 빈 응답을 출력했다. Qwen3.5 9B에서 2비트 양자화는 정보 손실이 너무 크다고 할 수 있다.

## 9. 양자화가 모델 속도에 미치는 영향

더 작은 양자화는 **더 빠른 모델**로 이어지는 경향이 있다. GPU 내부에서 이동시켜야 할 데이터가 적기 때문이다.

`llama.cpp`의 `llama-bench` 도구로 MacBook Pro M1 Max와 임대한 H100 SXM GPU에서 측정했다.

| 포맷 | M1 Max (tok/s) | H100 (tok/s) |
|------|---------------|-------------|
| bfloat16 | 19.45 | 106.85 |
| 8비트 대칭 | 32.36 | 141.61 |
| 4비트 비대칭 | 43.32 | 175.70 |
| 4비트 대칭 | 46.05 | 177.06 |
| 2비트 비대칭 | 40.25 | 166.90 |

bfloat16 대비 8비트와 4비트 양자화에서 큰 성능 향상이 있다. 2비트가 4비트보다 느린 이유는 저자도 확실하지 않다고 한다.

## 10. 결론

이 글에서 가장 중요한 핵심:

> **양자화된 모델은 꽤 괜찮다.**

저자도 처음에는 "모델 품질이 압축률에 비례해서 선형으로 떨어질 것"이라고 가정했다고 한다. bfloat16에서 8비트로 가면 절반, 4비트로 가면 또 절반... 이런 식으로.

**그렇지 않았다.**

- **16비트 → 8비트**: 품질 손실이 거의 없다
- **16비트 → 4비트**: 눈에 띄지만, 원본의 1/4 수준이 아니다. 측정 방법에 따라 대략 **원본의 90%** 수준
- **16비트 → 2비트**: 품질 절벽(quality cliff). 사용 불가 수준

로컬에서 양자화된 모델을 돌리는 것을 두려워하지 마라. 어딘가에 품질 절벽이 있지만, 이 글에서 그것을 식별할 도구를 배웠다. 절벽에서 떨어지지만 않았다면, 양자화된 모델은 잘 작동하고 압축되었다는 이유만으로 기피할 필요가 없다.

## 부록: 실행 명령어

### llama.cpp 설치
```bash
brew install llama.cpp
```

### Qwen 3.5 9B 다운로드
```bash
llama-server -hf unsloth/Qwen3.5-9B-GGUF:BF16 --port 8000
```

### 모델 양자화
```bash
cd ~/Library/Caches/llama.cpp
llama-quantize <원본파일>.gguf <출력파일>.gguf Q8_0
```

양자화 포맷 문자열:

| 포맷 | 문자열 |
|------|--------|
| 8비트 대칭 | Q8_0 |
| 4비트 비대칭 | Q4_1 |
| 4비트 대칭 | Q4_0 |
| 2비트 비대칭 | Q2_K |

### 퍼플렉서티 & KL 발산 측정
```bash
# 기준 데이터 생성
llama-perplexity -hf unsloth/Qwen3.5-9B-GGUF:BF16 \
  -f wikitext-2-raw/wiki.test.raw -c 512 \
  --kl-divergence-base ~/reference.kld

# 양자화 모델 측정
llama-perplexity -m <양자화모델>.gguf \
  -f wikitext-2-raw/wiki.test.raw -c 512 \
  --kl-divergence-base ~/reference.kld --kl-divergence
```

### 성능 벤치마크
```bash
llama-bench -hf unsloth/Qwen3.5-9B-GGUF:BF16
```

## 더 읽을거리

이 글은 **학습 후 양자화(Post-Training Quantization, PTQ)**만 다뤘다. 일부 모델은 **양자화 인식 학습(Quantization-Aware Training, QAT)**을 거치기도 하는데, 사전 학습 중에 양자화를 도입하여 파라미터가 양자화에 잘 맞게 설정되도록 한다.

양자화 외에도 LLM 크기를 줄이는 방법으로 **파라미터 가지치기(pruning)**와 **지식 증류(distillation)**가 있다. 더 복잡한 양자화 방법으로는 **AWQ**와 **GPTQ**가 있으니 관심 있다면 찾아보면 좋다.
